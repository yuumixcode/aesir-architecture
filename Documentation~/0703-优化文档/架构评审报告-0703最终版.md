## Aesir Architecture 架构评审报告（0703 最终版）

> 评审日期：2026-07-03
> 评审范围：`cn.runestone.aesir.architecture` 全部 C# 源码（~80 文件 / ~2805 LOC）
> 综合来源：QoderWork 评审、Codely 评审、RAA 深度审查（HTML）
> 状态：经讨论修订，已确认为最终优化方向

---

### 一、架构定位与设计决策

#### 1.1 最终架构定位

Aesir Architecture 是一个**通用 Unity 客户端开发框架**，以 UPM 包形式分发。核心架构本身是一个 Context 模块，另有 Aesir Modules UPM 包承载多个功能模块。

#### 1.2 关键设计决策（已确认）

**保留 Context\<T\> 泛型 CRTP。** 允许项目中同时存在多个 Context 节点（如 `GameContext`、`LobbyContext`），各自通过 `Context<T>.Interface` 访问独立单例。这是架构扩展性的基础。

**Context 不持有独立事件总线实例。** 所有模块通过全局 `MiniEventBus.Global` 通信。事件类型天然按命名空间隔离（战斗模块用 `DamageEvent`、网络模块用 `DisconnectedEvent`），不需要总线级别的上下文隔离。跨 Context 监听（UI 监听战斗事件）是游戏开发的正常需求。

**新增子模块概念。** Context 模块包含多种子模块角色：Model、Service、View、Controller、Presenter。单个类通过继承对应的角色接口（`IModel`、`IService` 等），标记自己在架构中的角色。

**Engine 层彻底脱离 MonoBehaviour。** 只使用纯 C#、`[Serializable]` 特性和 `[RuntimeInitializeOnLoadMethod]`（Unity 引擎级特性，无法脱离）。所有 MonoBehaviour 相关的逻辑（调试面板、生命周期触发器）隔离在 Component 层。

**AesirView 为推荐 View 基类。** `AesirView<T>` 自动使用 Odin 序列化（当 Odin Inspector 可用时），是 View 的首选基类。`MonoView<T>` 作为备选，适用于有 Odin 但不希望使用 Odin 序列化的场景。两者有意区分，不是重复设计。

**PlayerLoop 和 AESIR_ARCHITECTURE 宏是面向用户/外部包的基础设施。** PlayerLoop 子系统供框架使用者注册帧级回调，`AESIR_ARCHITECTURE` 宏定义供 Aesir Modules 等外部 UPM 包检测架构是否存在。框架自身不消费它们不是缺陷，而是设计意图。

### 二、确认的架构缺陷与优化方案

#### 缺陷 1：Engine → Component 层级倒置

**问题：** `AbstractContext<T>`（Engine 层，声称纯 C#）的 3 个方法直接调用 `ContextBoard.Instance.AddContext()`，触发 Component 层 MonoBehaviour 的创建。Engine 层无法脱离 Unity 运行时测试。

**位置：**
```
AbstractContext.cs → L31, L65, L93: ContextBoard.Instance.AddContext(instance)
ContextBoard.cs   → MonoBehaviour, 创建 GameObject
```

**方案：** 当前版本直接移除 `ContextBoard` 和 `MiniEventBusBoard`，同时移除 `AbstractContext` 中所有对 `ContextBoard.Instance.AddContext()` 的调用。Engine 层不再有任何 Component 层依赖，真正成为纯 C#。Editor 调试改为反射读取（见设计决策 1.2）。不需要引入 `IContextRegistry` 适配器——调试面板不存在，就没有注册目标。

#### 缺陷 2：AbstractModel 和 AbstractService 代码逐行相同

**问题：** 两者的 `_context` 字段、`Context` 属性、`SetContext`、`GetDependencies()`、`Initialized`、`Initialize()`、`Dispose()`、`OnInitialize()`、`OnDispose()` 完全一致。修改生命周期逻辑必须同步改两处。

**方案：** 提取 `AbstractSubmodule` 基类：

```csharp
public abstract class AbstractSubmodule : IContextHolder, ICanSetContext, ICanInitialize
{
    IContext _context;
    IContext IContextHolder.Context => _context;
    void ICanSetContext.SetContext(IContext context) => _context = context;

    public bool Initialized { get; private set; }
    public virtual HashSet<Type> GetDependencies() => null;

    void ICanInitialize.Initialize()
    {
        OnInitialize();
        Initialized = true;
    }

    public void Dispose()
    {
        OnDispose();
        _context = null;
    }

    protected abstract void OnInitialize();
    protected virtual void OnDispose() { }
}

public abstract class AbstractModel : AbstractSubmodule, IModel { }
public abstract class AbstractService : AbstractSubmodule, IService { }
```

#### 缺陷 3：Locator / ILocator 非泛型双轨——框架零引用

**问题：** `Locator`（140 行）和 `ILocator`（63 行）是 `GenericLocator<T>` 的非泛型翻版。框架内部只使用 `GenericLocator<IModel>`、`GenericLocator<IService>`、`GenericLocator<IContext>`。非泛型版本零调用。

**方案：** 直接删除 `Locator.cs` 和 `ILocator.cs`，净减约 200 行。YAGNI 原则——如果未来需要，到时再加。

#### 缺陷 4：能力接口文件碎片化

**问题：** 6 个 `ICan*` 标记接口各一个文件（每个仅 3 行有效代码），5 个 Extensions 类各一个文件（每个 8-25 行），共 11 个文件。扩展方法模式本身正确（写一次，所有实现类自动获得），但文件组织过于分散。

**方案：** 合并为 2 个文件。ICan 接口全部收入 `Capabilities.cs`，扩展方法收入 `CapabilityExtensions.cs`。逻辑零改动。

```csharp
// Capabilities.cs
public interface ICanGetModel : IContextHolder { }
public interface ICanGetService : IContextHolder { }
public interface ICanInvokeEvent : IContextHolder { }
public interface ICanExecuteCommand : IContextHolder { }
public interface ICanAddListener : IContextHolder { }

// CapabilityExtensions.cs
public static class CapabilityExtensions
{
    public static T GetModel<T>(this ICanGetModel self) where T : class, IModel =>
        self.Context.GetModel<T>();
    // ...其余方法集中于此
}
```

#### 缺陷 5：IService 定位模糊，文档与实现矛盾

**问题：** 注释称"Service 是主动调用的，不被动监听"，但接口实际继承了 `ICanAddListener`。同时 Service 还拥有 `ICanExecuteCommand`，但 Service 的核心定位是跨模块协调，不应直接执行 Command（Command 应由 Controller/Presenter 触发）。

**方案：** 明确 Service 为"万能子模块"定位——拥有除 `ICanExecuteCommand` 以外的所有能力：

```csharp
// Service 定位：万能协调层，能读写 Model、监听事件、发布事件，但不直接执行 Command
public interface IService : IContextHolder, ICanSetContext, ICanGetModel, ICanGetService,
    ICanInvokeEvent, ICanAddListener, ICanInitialize
{
    // 注意：不包含 ICanExecuteCommand
}
```

同步修正注释，去掉"不被动监听"的错误描述，改为准确描述 Service 的万能协调角色。

#### 缺陷 6：IView 的"只读"定位名不副实

**问题：** `IView` 被描述为"只读契约"，但实际继承了 `ICanInvokeEvent` 和 `ICanGetService`。一个能调用事件、能获取 Service 的接口不应被标记为"只读"。

**方案：** 移除"只读"描述，明确 IView 的定位——View 可以通过事件向上通信，可以读取 Model 和 Service，但不能执行 Command 或修改 Model 状态。

#### 缺陷 7：MiniEventBus.EventDictionary 是 public 暴露的

**问题：** 任何人都可以直接修改字典内容（Add/Remove/Clear），绕过 `AddListener`/`RemoveListener` 的封装逻辑和 `OnEventRegistrationsChanged` 事件触发。

**方案：** 改为 `private`，通过 `IReadOnlyDictionary<Type, Delegate>` 只读暴露给 Editor 调试面板（通过反射或专用接口）。

#### 缺陷 8：RemoveListenerOnSceneUnloadedTrigger 未注册域重载重置

**问题：** `GenericResetStaticsAssistant` 覆盖了 `GenericLocator`，但 `RemoveListenerOnSceneUnloadedTrigger._instance` 未注册重置回调。Disable Domain Reload 后该字段可能残留旧引用，导致访问已销毁的对象。（`ContextBoard._instance` 和 `MiniEventBusBoard._instance` 随当前版本移除这两个组件而不复存在。）

**方案：** 为 `RemoveListenerOnSceneUnloadedTrigger` 注册 `GenericResetStaticsAssistant.Register()` 回调，在域重载时置空 `_instance`。

#### 缺陷 9：依赖列表机制过于刚性——改用运行时 GetModel/GetService 错误日志

**问题：** 当前 Model 和 Service 都通过 `GetDependencies()` 声明依赖列表，Context 在初始化时校验依赖是否已注册并已初始化。这种方式假设依赖关系是静态固定的，但实际开发中 Model 经常需要在测试场景中被替换为可视化调试版本（通过场景中的 MonoBehaviour 在 Awake 时注册替换）。前置校验与运行时替换互相矛盾。

**方案：** 移除 `IModel.GetDependencies()` 和 `IService.GetDependencies()`，同时删除整个 `ContextDependencyAssistant` 类。改为在 `GetModel<T>()` 和 `GetService<T>()` 扩展方法中加入详细错误日志：

```csharp
public static T GetModel<T>(this ICanGetModel self) where T : class, IModel
{
    var model = self.Context.GetModel<T>();
    if (model == null)
    {
        throw new InvalidOperationException(
            $"{AesirArchitectureLog.ErrorTag} [{self.GetType().Name}] 尝试获取 Model [{typeof(T).Name}]，" +
            $"但该 Model 未在 Context 中注册。请确认已在 Configure() 中调用 RegisterModel<{typeof(T).Name}>()。");
    }
    return model;
}

public static T GetService<T>(this ICanGetService self) where T : class, IService
{
    var service = self.Context.GetService<T>();
    if (service == null)
    {
        throw new InvalidOperationException(
            $"{AesirArchitectureLog.ErrorTag} [{self.GetType().Name}] 尝试获取 Service [{typeof(T).Name}]，" +
            $"但该 Service 未在 Context 中注册。请确认已在 Configure() 中调用 RegisterService<{typeof(T).Name}>()。");
    }
    return service;
}
```

运行时效果：如果 `InventoryModel` 调用了 `GetModel<ItemDefinitionModel>()` 但后者未注册，报错信息为 `[InventoryModel] 尝试获取 Model [ItemDefinitionModel]，但该 Model 未在 Context 中注册`。同理 `GetService` 也会给出完整的调用者和目标类型信息——谁要什么、缺什么、去哪补，一行信息全有。

这种方式对"场景启动时替换 Model"的调试模式完全兼容：不关心注册顺序，不关心依赖声明，只在真正取不到时才报错。

### 三、子模块依赖管理策略

#### 移除 GetDependencies()——Model 和 Service 均不保留

原设计通过 `GetDependencies()` 声明模块间的依赖关系，Context 在初始化时做前置校验。这种方案有两个问题：

1. **过于刚性。** Model 经常需要在测试场景中被替换为可视化调试版本。例如在启动场景或测试场景中添加一个 MonoBehaviour 物体，在 `Awake()` 中将 `CombatModel` 替换为 `DebugCombatModel`（带 Inspector 可视化、可拖拽调试）。依赖列表假设依赖关系是静态固定的，与运行时替换矛盾。

2. **维护成本高于收益。** 开发者需要为每个 Model 手动维护 `HashSet<Type>` 依赖列表，而实际的依赖关系往往只在 `OnInitialize()` 中通过 `GetModel<T>()` 调用体现——依赖列表是对已有调用关系的冗余声明。

**最终方案：** 移除 `IModel` 和 `IService` 上的 `GetDependencies()` 方法，同时删除 `ContextDependencyAssistant` 整个类。改用运行时错误日志（见缺陷 9）作为安全保障。

#### Model 可视化调试模式

当需要可视化调试某个 Model 时：

1. 在启动场景或测试场景中添加一个 MonoBehaviour 调试物体
2. 在 `Awake()` 中调用 `Context.RegisterModel(new DebugXxxModel())` 替换目标 Model
3. Debug Model 可以将 `ObservableValue` 字段暴露为 `[SerializeField]`，支持 Inspector 实时查看和拖拽修改
4. 替换对其他模块透明——它们仍然通过 `GetModel<IXxxModel>()` 获取，无需任何修改

```csharp
public class ModelDebugBootstrap : MonoBehaviour
{
    void Awake()
    {
        // 替换 CombatModel 为带 Inspector 可视化的调试版本
        GameContext.Interface.RegisterModel(new DebugCombatModel());
    }
}
```

这种模式的核心优势：**调试时替换，上线时移除**，不侵入任何业务代码。

**前置条件建议：** 如果某个 Model 需要在 Inspector 中可视化调试，该 Model 实现类应标记 `[Serializable]`，使其 `ObservableValue<T>` 等数据字段能被 Unity 序列化器展示。`AbstractModel` 基类已标记 `[Serializable]`，继承基类的 Model 自动满足。不需要可视化的 Model（如持有网络连接、文件句柄等非序列化数据的 Model）无需标记。这不是架构的硬性要求，而是可视化调试模式的可选前提。

### 四、移除可视化监听组件

#### 当前版本移除的内容

当前版本直接移除以下 Editor 可视化监听组件：

- **ContextBoard**（`Runtime/Component/Core/ContextBoard.cs`）—— MonoBehaviour 组件，实时缓存 Context 的 Model/Service 列表
- **MiniEventBusBoard**（`Runtime/Component/Event/MiniEventBusBoard.cs`）—— MonoBehaviour 组件，实时监听事件总线注册变化
- **相关 Odin AttributeProcessor**（`ContextBoardAttributeProcessor`、`MiniEventBusBoardAttributeProcessor`）

#### 移除原因

这些组件在 Runtime 维护实时数据同步，导致：

1. Engine 层依赖 Component 层——`AbstractContext` 直接调用 `ContextBoard.Instance.AddContext()`，造成层级倒置
2. 每次 Context 变化都触发跨层调用，产生不必要的耦合
3. 占用 Runtime 内存维护调试用缓存，正式运行时完全浪费

#### 同步修改

- 移除 `AbstractContext` 中所有对 `ContextBoard.Instance.AddContext()` 的调用（3 处），Engine 层彻底脱离 Component 层
- 移除 `AesirArchitecture.Bootstrap()` 中对 `MiniEventBusBoard.Instance` 的初始化调用

#### 未来 Editor 调试方向

如果后续需要 Editor 调试面板，方向是通过反射按需读取 Context 内部状态（Model 列表、Service 列表）和 `MiniEventBus.EventDictionary`（事件注册表），不维护 Runtime 实时缓存，不引入任何 Runtime 监听组件。

### 五、Service 角色定位明确

Service 定位为**万能子模块**——拥有除 `ICanExecuteCommand` 以外的所有能力。Command 的执行入口应由 Controller/Presenter 层触发，Service 负责协调和编排。

| 能力 | 保留 | 理由 |
|------|------|------|
| GetModel | 是 | Service 需要读写 Model 数据来协调 |
| GetService | 是 | Service 可以调用其他 Service 完成跨模块协调 |
| InvokeEvent | 是 | Service 协调完成后发布业务事件 |
| AddListener | 是 | Service 可以监听事件，被动触发跨模块协调 |
| ExecuteCommand | **否** | Command 应由 Controller/Presenter 触发，Service 不直接执行 |

### 七、面试准备要点

**能解释清楚的决策：**
- 为什么保留 Context\<T\>：多 Context 节点支持（如游戏场景切换时新旧 Context 并存）
- 为什么全局事件总线：事件类型天然隔离，跨 Context 监听是游戏开发常态
- 为什么扩展方法而非基类方法：写一次所有角色自动获得，不产生重复代码
- 为什么移除依赖列表：Model 可在测试场景中被替换为可视化调试版本，依赖列表假设静态关系，与运行时替换矛盾；改用 GetModel 运行时错误日志更安全灵活

**能被追问的弱点：**
- Engine 层曾调用 ContextBoard.Instance 导致层级倒置（当前版本已通过移除 ContextBoard 修复）
- EventDictionary 是 public 的（应改为私有或只读暴露）
- `[Serializable]` 标注在持有 IContext 引用的类上，Unity 序列化器无法处理这些字段

**展示架构思维的回答：**
- "我的架构骨架是 Context 容器 + 能力分层 + 事件总线 + 可观察属性。核心约 1500 行，其余是 Editor 工具和调试辅助。"
- "我区分了 Engine（纯 C#）和 Component（MonoBehaviour 适配）两层。之前 Engine 层对 Component 有泄漏（ContextBoard 调用），当前版本已移除这些调试组件，Engine 层现在是真正的纯 C#。"
- "子模块通过接口标记角色：IModel 是数据层，IService 是协调层，ICommand 是写操作，IView 是只读展示层。每个角色有明确的能力边界。"
